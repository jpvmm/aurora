## User Constraints (from CONTEXT.md)

### Locked Decisions

### Server lifecycle
- Start principal deve ser **auto-start sob demanda** (quando um comando depende de inferência).
- Também deve existir comando manual explícito: **`aurora model start`**.
- Deve existir comando manual explícito de parada: **`aurora model stop`**.
- Escopo do servidor gerenciado é **global por usuário** (compartilhado entre terminais).
- Política de desligamento automático: **não desligar sozinho** (somente manual).
- Aurora deve exibir aviso claro quando o servidor permanecer ativo e como pará-lo.
- Se o usuário trocar de modelo com servidor ativo, Aurora deve **pedir confirmação** antes de reiniciar.
- Se já existir servidor externo `llama.cpp`, Aurora deve **perguntar ao usuário** se reutiliza ou não.
- Servidor iniciado pelo Aurora deve rodar em **background**.
- Em queda inesperada, Aurora tenta **auto-restart 1x** e avisa o usuário.
- Em conflito de porta, Aurora tenta **próxima porta automaticamente**.
- Se auto-start disparar sem modelo configurado, Aurora deve **pedir modelo e persistir**.
- O trigger de auto-start cobre **qualquer comando que dependa de inferência**.

### Model startup UX
- Usuário deve passar o modelo uma vez e Aurora deve persistir para uso futuro.
- Fluxo deve manter fricção mínima: informar modelo e usar, sem exigir gerenciamento manual constante de processo.

### Health/status contract
- Comandos oficiais de observabilidade da fase: **`aurora model status`** e **`aurora model health`**.
- `model status` deve mostrar resumo operacional: modelo ativo, endpoint, porta, PID/processo e uptime.
- `model health` deve fazer checagem ativa de endpoint + disponibilidade do modelo.
- Saída padrão deve ser legível para humanos, com opção estruturada via **`--json`**.

### Failure/recovery behavior
- Falhas de start devem retornar diagnóstico claro + comando exato de recuperação.
- Mensagens devem manter padrão prático e acionável em pt-BR.

### Claude's Discretion
- Definir conjunto exato de flags opcionais (`--json`, timeout, force, etc.) por comando.
- Definir formato textual final das mensagens de sucesso/erro/aviso mantendo consistência com o padrão já existente no CLI.
- Definir detalhes de priorização quando houver múltiplos endpoints candidatos disponíveis.

## Summary

This phase should be planned as runtime orchestration on top of existing Phase 1 primitives, not as a new runtime stack. The existing `RuntimeSettings`, `LlamaRuntimeClient`, and `doctor` diagnostic style are the foundation; Phase 01.1 adds lifecycle ownership (managed vs external server), process persistence across terminals, and explicit CLI observability commands.

The highest-risk planning areas are process ownership and cross-terminal consistency: deciding when Aurora "owns" a process, how to detect stale PID state, and how to avoid duplicate starts from concurrent terminals. Keep the implementation small and deterministic: one global state file, one lock, and clear transitions (`stopped -> starting -> running -> crashed`).

**Primary recommendation:** Plan around a single lifecycle service (`runtime/server_lifecycle.py`) consumed by CLI commands and future inference entrypoints, with explicit state persistence, lock protection, and reuse of `validate_runtime` for health gates.

## Architecture Patterns

### Recommended Project Structure

```text
src/aurora/
├── cli/
│   └── model.py                    # add start/stop/status/health
└── runtime/
    ├── server_lifecycle.py         # process orchestration + ownership
    ├── server_state.py             # dataclass + load/save state metadata
    ├── server_lock.py              # global lock guard (cross-terminal)
    ├── llama_client.py             # reuse health/model probe
    └── settings.py                 # existing endpoint/model persistence
```

### Pattern 1: Single Lifecycle Service (CLI-thin, runtime-thick)
**What:** Keep CLI commands as thin adapters; all lifecycle rules live in runtime module.
**When to use:** For `model start/stop/status/health` and future auto-start hooks.
**Example:**
```python
# cli/model.py
@model_app.command("start")
def model_start_command(...):
    result = ensure_server_running(interactive=True)
    render_start_result(result)
```

### Pattern 2: Managed vs External Ownership
**What:** Persist `ownership = managed|external` in lifecycle state.
**When to use:** Whenever endpoint is already live before a start request.
**Example:**
```python
@dataclass
class ServerState:
    ownership: Literal["managed", "external"]
    pid: int | None
    endpoint_url: str
    model_id: str
    started_at: str
```

### Pattern 3: Start Gate with Port Fallback + Readiness Loop
**What:** On start, try configured port then next ports; verify readiness through `/health` and `/v1/models`.
**When to use:** manual start and auto-start-on-demand.
**Example:**
```python
for port in candidate_ports(base_port, attempts=5):
    proc = launch_llama_server(port=port, alias=settings.model_id)
    if wait_until_ready(port=port, model_id=settings.model_id, timeout=60):
        return running_state(proc.pid, port)
    if process_exited(proc):
        continue
```

### Pattern 4: Auto-start Wrapper for Inference-Dependent Commands
**What:** Central `ensure_runtime_for_inference()` called before any inference operation.
**When to use:** every command path that invokes model inference.
**Example:**
```python
def ensure_runtime_for_inference() -> RuntimeSettings:
    settings = load_settings()
    ensure_model_configured(settings)
    ensure_server_running(interactive=is_tty())
    return load_settings()  # may include updated port
```

### Anti-Patterns to Avoid
- Parsing `ps`/`lsof` shell output for lifecycle truth; use persisted state + probe.
- Embedding lifecycle logic directly in Typer command functions.
- Returning only textual status without machine-readable `--json` output.
- Assuming model ID equals GGUF filename; explicitly pass `--alias` on server start.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| Health readiness logic | Custom ad-hoc HTTP checks in each command | Existing `validate_runtime()` + `LlamaRuntimeClient` | Already tested and provides typed diagnostics. |
| Daemonization shell tricks (`&`, `nohup`) | Shell-dependent process detaching | `subprocess.Popen(..., start_new_session=True)` | Portable Python-managed lifecycle. |
| Endpoint/model response parsing | Repeated ad-hoc JSON parsing across CLI | `LlamaRuntimeClient` return model IDs + errors | Consistent behavior and easier tests. |
| Manual model id matching guesswork | Infer id from filename/path heuristics | Start with `--alias <settings.model_id>` and verify `/v1/models` | Avoids mismatch between configured model and runtime ID. |

## Common Pitfalls

### Pitfall 1: Stale PID state breaks stop/status
**What goes wrong:** `status` shows running but PID no longer exists.
**Why it happens:** abrupt process death without state cleanup.
**How to avoid:** validate PID liveness on every `status/start/stop`; if dead, mark state as crashed and clear managed PID.
**Warning signs:** repeated "PID not found" or health failures with lingering state file.

### Pitfall 2: Duplicate starts from multiple terminals
**What goes wrong:** two managed servers race and conflict on port.
**Why it happens:** no cross-terminal lock.
**How to avoid:** global lock around start/stop transitions.
**Warning signs:** intermittent port conflict despite recent successful start.

### Pitfall 3: Model mismatch after `model set`
**What goes wrong:** runtime serves previous model while settings changed.
**Why it happens:** model changed without restart confirmation.
**How to avoid:** detect active server + changed model and require explicit confirmation before restart.
**Warning signs:** `model health` fails with `model_missing` immediately after model update.

### Pitfall 4: External server reuse ambiguity
**What goes wrong:** user cannot tell if `aurora model stop` will kill external process.
**Why it happens:** ownership not tracked.
**How to avoid:** status output must always display ownership (`managed` vs `external`) and `stop` behavior must respect it.
**Warning signs:** fear-driven UX: users avoid stop command.

### Pitfall 5: Noisy/opaque startup failures
**What goes wrong:** start fails without actionable recovery.
**Why it happens:** missing capture of startup stderr and binary resolution checks.
**How to avoid:** classify startup errors (`binary_missing`, `port_conflict`, `timeout`, `invalid_model`) with exact recovery commands.
**Warning signs:** user repeats `start` without understanding failure cause.

```python
proc = subprocess.Popen(
    command,

    stderr=stderr_log,
    start_new_session=True,
```

### Graceful stop then force kill if needed
```python
proc.terminate()
try:
    proc.wait(timeout=10)
except subprocess.TimeoutExpired:
    proc.kill()
```

## Open Questions

1. **How should Aurora find `llama-server` binary?**
- What we know: Phase requires start/stop from CLI, but binary install path is not defined in current project.
- What's unclear: strict requirement (`PATH`, configurable env var, or setup-time persisted path).
- Recommendation: plan explicit resolution order now (`AURORA_LLAMA_SERVER_BIN` -> `shutil.which("llama-server")`) with actionable error.

2. **How far should automatic port fallback go?**
- What we know: context requires "next port automatically".
- What's unclear: number of retries and whether endpoint setting should be mutated permanently.
- Recommendation: cap attempts (for example 5), persist final chosen endpoint, and display recovery guidance when exhausted.

3. **What is non-interactive behavior for confirmation prompts?**
- What we know: model switch and external reuse require confirmation.
- What's unclear: CI/scripts without TTY cannot answer prompts.
- Recommendation: include deterministic flags in plan (`--yes`/`--force`) and fail fast without interaction when omitted.

4. **Where should auto-start be wired today?**
- What we know: trigger must cover any inference-dependent command.
- What's unclear: current codebase has no production `ask` command yet.
- Recommendation: deliver shared auto-start guard in runtime this phase and integrate immediately where inference already occurs; keep extension hook for upcoming `ask` command.
