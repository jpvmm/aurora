# Phase 02.1: KB UX + Auto-Embeddings - Context

**Gathered:** 2026-03-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Make the Aurora KB suite easier to operate by simplifying collection configuration and update/delete workflows, while removing the need for users to run qmd embedding commands manually.

This phase clarifies product behavior for KB operations and automation. It does not add unrelated retrieval/memory capabilities.

</domain>

<decisions>
## Implementation Decisions

### Auto-Embeddings Behavior
- Auto-embeddings run when there is actual KB change (delta), not blindly on every command.
- Execution mode is blocking by default (command finishes only after embeddings step completes).
- Embedding failure is treated as partial failure: KB text/index state may complete, but user must receive clear warning and recovery command.
- Auto-embedding behavior is controlled by global Aurora config (not by requiring user to run qmd manually).

### Collection Configuration UX
- Introduce dedicated `aurora kb config` command surface for KB settings.
- Keep one active global collection/index at a time (single active target model).
- `update/delete/rebuild` use active global target by default, with optional per-command override.
- `aurora config show` must display KB-relevant settings (vault/index/collection/auto-embeddings), with `aurora kb config show` as detailed KB view.

### Update/Delete UX
- `aurora kb delete` default behavior remains full cleanup of active collection scope.
- Delete safety: interactive confirmation + explicit `--yes` for non-interactive/destructive flow.
- Update flow must support periodic execution configured by user.
- Output should be concise and readable, with next-step guidance/recovery commands.

### Scheduler and Operational Control
- Scheduler is OFF by default; user opts in explicitly.
- Scheduler interval policy: daily execution at fixed local time (user-defined hour).
- Provide explicit lifecycle commands for scheduler control: enable/disable/status.
- Concurrency policy: global KB lock with deterministic queue/skip behavior (no parallel KB mutations).
- Missed run policy: if machine is OFF at scheduled time, run one catch-up on next boot, then return to daily fixed schedule.

### Claude's Discretion
- Exact naming and grouping of CLI flags/options, as long as behavior above is preserved.
- Retry/backoff tuning for embedding step failures.
- Final formatting details for human-readable progress and warning output.

</decisions>

<specifics>
## Specific Ideas

- User expectation: KB maintenance should feel automatic and simple, without manual qmd embedding commands.
- Periodic update expectation: "everyday" with fixed hour and robust behavior if computer is powered off.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/aurora/cli/kb.py`: existing KB command namespace (`ingest/update/delete/rebuild`) and shared text/json rendering patterns.
- `src/aurora/runtime/settings.py`: persisted global settings already include KB fields (`kb_vault_path`, `kb_qmd_index_name`, `kb_qmd_collection_name`).
- `src/aurora/kb/service.py`: deterministic lifecycle orchestration and error contracts (`KBServiceError`, progress stages).
- `src/aurora/kb/qmd_backend.py`: concrete QMD transport with collection bootstrap/refresh and typed diagnostics.

### Established Patterns
- Aurora uses global persisted config (`RuntimeSettings`) with strict validation and PT-BR diagnostics.
- CLI surfaces use Typer command groups and deterministic text/json output contracts.
- Safety-oriented guardrails are already used in runtime lifecycle commands (`--yes`/confirmation patterns).

### Integration Points
- Add new KB configuration/scheduler command surface under `src/aurora/cli/kb.py` (or sibling module wired under `kb`).
- Extend `RuntimeSettings` and `aurora config show` to expose KB operational settings.
- Extend `KBService` orchestration to trigger/track embedding phase and scheduler-driven updates consistently.
- Reuse existing lock/ownership patterns from runtime lifecycle for KB scheduler concurrency control.

</code_context>

<deferred>
## Deferred Ideas

- Multi-profile/multi-collection active contexts (e.g., work/personal switching).
- Selective delete by path/glob as first-class UX (instead of full active-scope delete).

</deferred>
