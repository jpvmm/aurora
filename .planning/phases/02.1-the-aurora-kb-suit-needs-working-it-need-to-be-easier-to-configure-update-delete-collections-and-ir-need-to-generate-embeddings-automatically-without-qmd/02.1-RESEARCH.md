## User Constraints (from CONTEXT.md)

### Locked Decisions

### Auto-Embeddings Behavior
- Auto-embeddings run when there is actual KB change (delta), not blindly on every command.
- Execution mode is blocking by default (command finishes only after embeddings step completes).
- Embedding failure is treated as partial failure: KB text/index state may complete, but user must receive clear warning and recovery command.
- Auto-embedding behavior is controlled by global Aurora config (not by requiring user to run qmd manually).

### Collection Configuration UX
- Introduce dedicated `aurora kb config` command surface for KB settings.
- Keep one active global collection/index at a time (single active target model).
- `update/delete/rebuild` use active global target by default, with optional per-command override.
- `aurora config show` must display KB-relevant settings (vault/index/collection/auto-embeddings), with `aurora kb config show` as detailed KB view.

### Update/Delete UX
- `aurora kb delete` default behavior remains full cleanup of active collection scope.
- Delete safety: interactive confirmation + explicit `--yes` for non-interactive/destructive flow.
- Update flow must support periodic execution configured by user.
- Output should be concise and readable, with next-step guidance/recovery commands.

### Scheduler and Operational Control
- Scheduler is OFF by default; user opts in explicitly.
- Scheduler interval policy: daily execution at fixed local time (user-defined hour).
- Provide explicit lifecycle commands for scheduler control: enable/disable/status.
- Concurrency policy: global KB lock with deterministic queue/skip behavior (no parallel KB mutations).
- Missed run policy: if machine is OFF at scheduled time, run one catch-up on next boot, then return to daily fixed schedule.

### Claude's Discretion
- Exact naming and grouping of CLI flags/options, as long as behavior above is preserved.
- Retry/backoff tuning for embedding step failures.
- Final formatting details for human-readable progress and warning output.

### Deferred Ideas (OUT OF SCOPE)

- Multi-profile/multi-collection active contexts (e.g., work/personal switching).
- Selective delete by path/glob as first-class UX (instead of full active-scope delete).

## Summary

Aurora already has a solid KB lifecycle core (`KBService` + `QMDAdapter` + `QMDCliBackend`) and deterministic CLI contracts, but it currently lacks three things this phase needs: a KB-focused config UX, automatic embedding orchestration, and a scheduler runtime. The current backend only runs `qmd update`; no embedding command is executed by Aurora today.

The best planning approach is to keep lifecycle ownership in Aurora and extend the existing service/contract pattern instead of wiring behavior directly in CLI handlers. That means: (1) add KB config/scheduler state to global settings + KB state file, (2) extend backend protocol with an explicit embedding step, (3) return structured operation results that can express partial failure cleanly, and (4) introduce a global KB mutation lock mirroring server lifecycle lock behavior.

**Primary recommendation:** implement one `KBOrchestrator` pipeline (`scan -> delta -> apply -> embed-if-needed -> summary`) and one `KBSchedulerService` (`enable/disable/status/run-due`) with explicit locking and persisted state.

## Architecture Patterns

### Recommended Project Structure

```text
src/aurora/
|-- cli/
|   |-- kb.py                     # kb ingest/update/delete/rebuild + new subcommands
|   `-- config.py                 # extend `aurora config show` KB summary
|-- kb/
|   |-- service.py                # extend operation result for partial embedding failures
|   |-- qmd_backend.py            # add embed command transport
|   |-- qmd_adapter.py            # keep manifest mutation semantics deterministic
|   |-- scheduler.py              # new scheduler service (enable/disable/status/run-due)
|   `-- scheduler_state.py        # persisted KB scheduler state (new)
`-- runtime/
    |-- settings.py               # add KB config fields (auto-embeddings + schedule config)
    `-- paths.py                  # add scheduler lock/state paths if needed
```

### Pattern 1: Operation Envelope With Partial-Failure Contract
**What:** Keep one typed operation result that includes `index_result` and `embedding_result`.
**When to use:** `ingest`, `update`, `delete`, `rebuild`.
**Example:**
```python
@dataclass(frozen=True)
class KBEmbeddingStatus:
    attempted: bool
    ok: bool
    category: str | None
    recovery_command: str | None

# embed only when delta mutated index state and auto-embedding enabled
if state_mutated and settings.kb_auto_embeddings_enabled:
    embedding_status = backend.embed()
```

### Pattern 2: Delta-Gated Embedding
**What:** Execute embedding only when operation changed indexed scope (`added/updated/removed > 0` or adapter state mutated).
**When to use:** all mutating KB commands.
**Why:** Aligns with locked decision to avoid blind embedding runs.

### Pattern 3: CLI Safety Pattern Reuse (`--yes` + interactive confirm)
**What:** Mirror existing model command pattern for destructive actions.
**When to use:** `aurora kb delete` and potentially scheduler disable/reset paths.
**Example:**
```python
if not yes:
    confirmed = typer.confirm("Remover toda a colecao ativa?", default=False)
    if not confirmed:
        raise typer.Exit(code=1)
```

### Pattern 4: Global KB Mutation Lock
**What:** Introduce a KB lock around `update/delete/rebuild/scheduler-triggered update`.
**When to use:** every operation that mutates KB state/index.
**Why:** Prevent race conditions between manual and scheduled runs.

### Pattern 5: Scheduler as `run-due` Engine + Control Commands
**What:** Scheduler control commands only toggle config/state; execution logic lives in a deterministic `run_due(now)` path.
**When to use:** `enable/disable/status` and boot/daily triggers.
**Why:** keeps testability high and isolates time-based logic.

### Anti-Patterns to Avoid
- Running embed unconditionally on every command.
- Coupling embed command execution directly inside Typer handlers.
- Returning success without exposing embedding partial-failure warnings/recovery commands.
- Implementing scheduler without lock coordination against manual `aurora kb update`.
- Adding multi-collection profile concepts in this phase (explicitly deferred).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---|---|---|---|
| CLI safety UX | Custom ad-hoc prompt flow | Existing Typer `confirm` + `--yes` non-interactive override pattern | Already proven in `model` commands/tests. |
| Concurrency control | Homemade per-command mutex behavior | Existing Aurora file-lock architecture pattern | Deterministic cross-terminal behavior already established. |
| Embedding orchestration semantics | Separate manual user workflow | Service-level automatic embed step after successful delta application | Matches locked UX and keeps logic centralized. |
| Schedule parsing | Custom cron parser | Fixed daily hour config (`0..23`) + due-check computation | Smaller, deterministic, easier to test than cron DSL. |

**Key insight:** This phase is primarily orchestration and UX contracts, not new retrieval algorithms.

## Common Pitfalls

### Pitfall 1: Embedding Failure Ambiguity
**What goes wrong:** command exits as success even when embeddings fail; user assumes retrieval freshness.
**Why it happens:** no dedicated embedding status channel.
**How to avoid:** explicit partial-failure status + warning + exact recovery command (`aurora kb update --force-embed` or equivalent).
**Warning signs:** logs show update done but no embed stage outcome.

### Pitfall 2: Delete Without Confirmation in Non-Interactive Context
**What goes wrong:** scripted execution wipes active collection unintentionally.
**Why it happens:** missing `--yes` enforcement.
**How to avoid:** fail fast in non-interactive mode unless explicit `--yes`.
**Warning signs:** CI or scripts executing `aurora kb delete` with no safety gate.

### Pitfall 3: Settings Drift Between `config show` and `kb config show`
**What goes wrong:** two commands report inconsistent active vault/index/collection settings.
**Why it happens:** duplicated rendering logic.
**How to avoid:** centralized KB settings serializer reused by both commands.
**Warning signs:** tests for two outputs diverge after setting changes.

### Pitfall 4: Scheduler Race With Manual Update
**What goes wrong:** concurrent mutation corrupts operator expectations and can misreport counters.
**Why it happens:** scheduler and CLI run without shared lock.
**How to avoid:** enforce one global KB lock for all mutating paths.
**Warning signs:** intermittent `state_mismatch`/divergence after overlapping runs.

### Pitfall 5: DST/Local-Time Misfires
**What goes wrong:** daily job runs twice or skips after daylight saving transition.
**Why it happens:** naive datetime handling.
**How to avoid:** store timestamps in UTC, compute next due from configured local hour with timezone-aware conversion, and persist last successful scheduled slot.
**Warning signs:** schedule status jumps unexpectedly around timezone transitions.

## Open Questions

1. **Scheduler platform scope for boot catch-up**
- What we know: requirement asks for catch-up on next boot.
- What's unclear: whether this phase must support macOS/Linux/Windows equally or can target current primary OS first.
- Recommendation: lock scope before planning tasks; if cross-platform is required, split scheduler adapters by OS behind one service contract.

2. **Partial failure exit code policy**
- What we know: embedding failure must be surfaced clearly as partial failure.
- What's unclear: should CLI exit non-zero on partial failure or zero with warning metadata.
- Recommendation: define one policy early and enforce across text and `--json` outputs/tests.

3. **Per-command override design**
- What we know: update/delete/rebuild should use active global target by default but allow override.
- What's unclear: exact override flags (`--index`, `--collection`, temp-only vs persisted).
- Recommendation: choose non-persisting per-command override to avoid silent config mutation.
