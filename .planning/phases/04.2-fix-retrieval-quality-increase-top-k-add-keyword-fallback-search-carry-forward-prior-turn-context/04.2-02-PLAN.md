  <behavior>
    Carry-forward state (D-08, D-09, D-10):
    - ChatSession initializes _last_retrieved_paths: list[str] = []
    - After _handle_vault_turn completes, _last_retrieved_paths is updated with the paths from the retrieval result (up to 3, per Claude's discretion)
    - After _handle_memory_turn completes, _last_retrieved_paths is also updated
    - After _handle_chat_turn completes, _last_retrieved_paths is cleared to [] (D-11: no carry-forward for chat)

    Carry-forward application (D-09):
    - When _handle_vault_turn runs and _last_retrieved_paths is non-empty:
      - After fresh retrieval, check which carry-forward paths are NOT already in fresh results
      - For missing paths, fetch content via self._retrieval._backend.fetch(path)
      - Append as supplementary RetrievedNote objects (score=0.0, source matches original) after fresh notes
      - Cap carry-forward supplements at 3 notes max
      - Rebuild context_text with the combined notes
    - When _handle_memory_turn runs, same carry-forward logic applies

    Edge cases:
    - First turn (no previous context): _last_retrieved_paths is empty, no carry-forward runs
    - Carry-forward path fetch returns None: skip that path silently
    - Fresh results already include all carry-forward paths: no supplements added
    - Insufficient evidence from fresh search: no carry-forward attempted (nothing to supplement)
  </behavior>
  <action>
    **ChatSession state (per D-08, D-09, D-10):**
    1. In `__init__`, add: `self._last_retrieved_paths: list[str] = []`

    **Carry-forward helper (per D-09):**
    2. Add a private method `_apply_carry_forward(self, result: RetrievalResult) -> RetrievalResult`:
       - If `self._last_retrieved_paths` is empty, return `result` unchanged.
       - If `result.insufficient_evidence` is True, return `result` unchanged (no carry-forward on empty).
       - Compute `fresh_paths = {n.path for n in result.notes}`.
       - Compute `missing_paths = [p for p in self._last_retrieved_paths if p not in fresh_paths][:3]` (cap at 3).
       - For each missing path, call `self._retrieval._backend.fetch(path)`:
         - If content is None, skip.
         - Otherwise create `RetrievedNote(path=path, score=0.0, content=content, source="vault")`.
       - Combine: `all_notes = list(result.notes) + supplements`.
       - Rebuild context_text: call `self._retrieval._assemble_context(all_notes)`.
       - Return new `RetrievalResult(ok=True, notes=tuple(all_notes), context_text=context_text, insufficient_evidence=False)`.

    **Update _handle_vault_turn (per D-08, D-09):**
    3. After retrieval (after `result = self._retrieval.retrieve(...)` or `retrieve_with_memory(...)`), BEFORE the insufficient_evidence check:
       - Apply carry-forward: `result = self._apply_carry_forward(result)`.
    4. After the full turn completes (after generating the response, before returning):
       - Update carry-forward state: `self._last_retrieved_paths = [n.path for n in result.notes][:3]`
       - This must happen even if the result was insufficient (in that case, paths will be empty since result.notes is empty tuple).

    **Update _handle_memory_turn (per D-08, D-11):**
    5. Same pattern as vault turn: apply carry-forward before insufficient check, update paths after response.

    **Update _handle_chat_turn (per D-11):**
    6. At the end of `_handle_chat_turn`, clear carry-forward: `self._last_retrieved_paths = []`.

    **Update process_turn:**
    7. No changes needed to process_turn itself -- the carry-forward state management happens inside each handler.

    **Tests (new class TestCarryForward in tests/chat/test_session.py):**
    8. Add test class `TestCarryForward`:

       - `test_vault_turn_stores_retrieved_paths`: Process a vault turn with 2 notes. Assert `session._last_retrieved_paths` contains those 2 paths.

       - `test_carry_forward_supplements_fresh_results`: Process vault turn 1 (returns note A). Then process vault turn 2 (returns note B, NOT note A). Assert turn 2's result includes both note B (fresh) and note A (carry-forward). Verify by checking that `_retrieval._backend.fetch` was called with note A's path.

       - `test_carry_forward_skips_paths_already_in_fresh_results`: Process vault turn 1 (returns note A). Process vault turn 2 (also returns note A). Assert `_retrieval._backend.fetch` was NOT called for carry-forward (note A already fresh).

       - `test_carry_forward_cleared_on_chat_intent`: Process vault turn (stores paths). Then process chat turn. Assert `session._last_retrieved_paths == []`.

       - `test_carry_forward_caps_at_3_paths`: Process vault turn with 5 notes. Assert `len(session._last_retrieved_paths) <= 3`.

       - `test_no_carry_forward_on_first_turn`: Process first vault turn. Assert no extra fetch calls beyond the normal retrieval flow.

       - `test_carry_forward_skips_when_fetch_returns_none`: Process vault turn 1 (returns note A). Set up `_backend.fetch` to return None for note A's path. Process vault turn 2 (returns note B). Assert no supplement added (carry-forward fetch failed silently).

       For mock setup in carry-forward tests: The mock retrieval service needs `_backend.fetch` accessible. Use `mock_retrieval._backend = MagicMock()` and configure `mock_retrieval._backend.fetch.side_effect` to return content for specific paths. Also mock `mock_retrieval._assemble_context` to return a string. The _apply_carry_forward method calls `self._retrieval._backend.fetch` and `self._retrieval._assemble_context` directly.
