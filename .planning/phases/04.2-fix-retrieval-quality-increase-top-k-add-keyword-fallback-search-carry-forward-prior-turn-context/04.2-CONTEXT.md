# Phase 04.2: Fix Retrieval Quality - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix retrieval quality so Aurora finds relevant notes for name queries, date-range queries, and follow-up questions. Three concrete fixes: higher top-K, keyword fallback search, and session context carry-forward.

</domain>

<decisions>
## Implementation Decisions

### Increase Top-K
- **D-01:** Increase `retrieval_top_k` default from 7 to 15. Broader date-range queries ("notas de março") need more results.
- **D-02:** Widen the validator range from 5-10 to 5-30. Let users tune it higher for large vaults.
- **D-03:** Increase `MAX_CONTEXT_CHARS` from 12,000 to 24,000 to accommodate more notes without truncating. Local models with 8K+ context can handle this.

### Keyword Fallback Search
- **D-04:** When `qmd query` (hybrid) returns results but the user's query contains proper nouns, names, or quoted terms — also run `qmd search` (BM25 keyword-only) as a fallback and merge results.
- **D-05:** Add `keyword_search()` method to `QMDSearchBackend` that calls `qmd search --json` instead of `qmd query --json`. Same collection, same index.
- **D-06:** In `RetrievalService.retrieve()`: after hybrid search, extract likely names/proper nouns from the query (capitalized words, quoted phrases). If found, run keyword search and merge results (dedup by path, keep highest score).
- **D-07:** Keyword search uses a lower min_score (0.1) since BM25 scores differently than hybrid.

### Session Context Carry-Forward
- **D-08:** `ChatSession` tracks retrieved note paths from the previous turn. On follow-up questions, these paths are passed as "supplementary context" alongside fresh search results.
- **D-09:** Implementation: after each vault/memory turn, store `last_retrieved_paths: list[str]`. On next turn, if fresh search doesn't include these paths, fetch them anyway and append (lower priority than fresh results).
- **D-10:** Carry-forward only works for the immediately previous turn (not full history). Keeps it simple and avoids context bloat.
- **D-11:** Only carry forward for vault and memory intents, not chat intent.

### Claude's Discretion
- Exact heuristic for detecting proper nouns (simple: words starting with uppercase that aren't sentence-start)
- Whether to run keyword search in parallel or sequentially after hybrid
- How many carry-forward notes to include (suggest max 3)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `QMDSearchBackend.search()` — shells out to `qmd query --json`. Can be mirrored for `qmd search --json`.
- `RetrievalService._dedup_hits()` — already deduplicates by path. Can merge keyword results with hybrid results.

### Key Facts
- `qmd search "Rosely" --json` finds exact matches that `qmd query` misses
- `qmd search` uses BM25 keyword matching, no embeddings needed
- `qmd query` uses hybrid (BM25 + vector + LLM rerank) — better for natural language but weaker for exact names

### Integration Points
- `QMDSearchBackend` — add `keyword_search()` method
- `RetrievalService.retrieve()` — add keyword fallback after hybrid
- `ChatSession` — add `_last_retrieved_paths` tracking

</code_context>

<specifics>
## Specific Ideas

- The keyword fallback is NOT a replacement for hybrid search. It's an "also check" for queries containing names/proper nouns.
- Carry-forward solves: "find Rosely" → found in diary → "what else did I write about her?" → diary note still in context

</specifics>
